Strands vs LangGraph — 그리고 그 밖의 것들

핵심 갈림길은 하나입니다: 모델이 루프를 결정하게 둘 것인가(Strands 단일 Agent), 아니면 개발자가 상태 그래프를 명시할 것인가(LangGraph). 재밌게도 Strands는 둘 다 제공합니다.

모델이 결정하나, 개발자가 그리나

같은 ReAct 작업이라도 프레임워크의 기본 추상화 레벨이 다릅니다. 왼쪽은 루프를 모델에 맡기고(고수준), 오른쪽은 노드·엣지·상태를 손으로 배선합니다(저수준·명시적).

Strands — Agentic Loop
model-driven · 고수준
  • 개발자는 툴만 준다. reason→act→observe 루프는 모델이 자율적으로 돈다.
  • 비결정적 경로 — 실행마다 다를 수 있음. 개방형 작업에 강함.
  • 더 구조가 필요하면 GraphBuilder(결정적 DAG)·Swarm으로 확장.
  • 비슷한 계열: OpenAI Agents SDK(handoff 중심의 얇은 루프).
LangGraph — State Graph
developer-defined · 저수준
  • 개발자가 그래프를 배선한다. 노드(함수)·엣지·조건분기를 명시.
  • 결정적·명시적 제어 — 분기·루프·중단/재개를 정밀하게.
  • 공유 State(TypedDict + 리듀서)가 노드 사이를 흐른다.
  • 빠른 시작은 프리빌트 create_agent(내부적으로 StateGraph).
한 줄 요약 — "루프를 모델에 맡길수록" 고수준·단순·비결정적(Strands Agent, OpenAI Agents), "그래프를 손으로 그릴수록" 저수준·명시적·결정적(LangGraph). Strands의 GraphBuilder와 LangGraph의 create_agent가 그 중간에서 서로를 향해 다가옵니다.

같은 ReAct 에이전트, 두 가지 코드

계산기 + 날씨 툴을 가진 동일한 에이전트를 각각으로 작성. 줄 수보다 무엇을 개발자가 쓰고 무엇을 프레임워크가 숨기는지를 보세요.

Strandsagentic loopfrom strands import Agent, tool
from strands_tools import calculator

@tool
def weather(city: str) -> str:
    "도시의 날씨를 반환한다."
    return f"{city}: 맑음 22도"

agent = Agent(tools=[calculator, weather])
agent("서울 날씨 알려주고 화씨로 바꿔줘")
# 루프는 모델이 알아서 — 개발자는 툴만 준다

from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[list, add_messages]  # 리듀서

차이의 본질 — Strands는 상태(messages)·루프를 프레임워크가 관리하고 개발자는 툴/노드만 얹습니다. LangGraph는 상태 스키마·리듀서·엣지·조건을 개발자가 1급 시민으로 다룹니다 — 그만큼 제어력이 크고, 그만큼 배선이 많습니다.

⚠️ 검증 시점(2026-07) 주의사항

  • LangGraph create_react_agent는 deprecated → from langchain.agents import create_agent 권장.
  • LangGraph 체크포인터 정식명은 InMemorySaver (MemorySaver는 레거시 별칭). 문서는 docs.langchain.com으로 이전됨.
  • Microsoft AutoGen은 유지보수 모드 — 신규 개발은 Microsoft Agent Framework(AutoGen+Semantic Kernel 통합). 오픈·자체호스팅 계보는 AG2.
  • Google ADK 2.0은 breaking 릴리스(그래프 Workflow 런타임 추가). 위 클래식 API는 유지되나 튜토리얼이 옛것일 수 있음.

언제 무엇을 고르나

정답은 없고, 제약(생태계·팀·제어 요구)에 따라 갈립니다.

Strands AWS

이미 AWS/Bedrock에 있고, "툴만 주고 모델에 맡기는" 단순함을 원할 때. 필요하면 Graph/Swarm으로 구조화. 배포·메모리·관측을 AgentCore로 일괄 처리.

LangGraph 벤더중립

제어 흐름을 손으로 정밀하게 그리고 싶을 때 — 복잡한 분기·루프, 중단/재개(HITL), 명시적 상태 관리가 핵심. 러닝커브 대신 최대 제어력.

CrewAI 역할기반

역할(role/goal/backstory) 기반 팀 협업을 빠르게 세우고 싶을 때. 높은 추상화 + 이벤트 기반 Flows로 흐름 제어.

OpenAI Agents SDK OpenAI

OpenAI 중심에서 가볍게 — Agents/Handoffs/Guardrails/Sessions 몇 개 프리미티브로 얇은 루프. 최소 추상화 선호 시.

Google ADK Google

Gemini/Vertex 중심에서, LLM 에이전트 위에 결정적 워크플로(Sequential/Parallel/Loop)를 얹고 Vertex로 배포할 때.

공통 원칙

대부분은 단일 에이전트 + 좋은 툴로 충분합니다. 멀티에이전트·복잡한 그래프는 정말 필요할 때 도입하세요 — 제어력은 늘지만 디버깅·비용도 늘어납니다.

큰 그림 — 프레임워크는 달라도 안에서 도는 건 같은 agentic loop이고(우리가 본 03번 루프), 여러 개를 엮는 방식(Graph·Swarm·handoff)과 상태·메모리·배포를 각자 다르게 포장할 뿐입니다. Strands를 이해했다면 나머지는 용어 매핑 문제에 가깝습니다.
